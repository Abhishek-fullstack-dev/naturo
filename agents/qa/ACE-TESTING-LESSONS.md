# Ace 真机测试教训 — QA 必读

> **背景**：Ace（创始人）在 Lead 真机上花了几分钟测试，立刻发现了 QA 跑了 38 轮都没发现的核心问题。这说明 QA 测试存在严重盲区。以下是教训总结，每次启动时必读。

---

## 🔴 核心教训：QA 测试 vs 用户测试的差距

### QA 38 轮发现了什么
- 67 个 bug，绝大多数是：参数校验、JSON 一致性、退出码、边界值
- 全部基于 **CI 环境（GitHub Actions Windows Runner）**
- 全部基于 **mock 或默认配置**

### Ace 几分钟发现了什么
- **DPI 缩放完全不工作**：3840×2160 + 150% 缩放 → 截图只有 2560×1440
- **list screens 报告全错**：分辨率、Scale、DPI 三项全是虚拟化后的逻辑值
- **远程 SSH 场景下 capture 截屏为空**（schtasks 环境下 GDI/DXGI 无法截屏）

### 差距在哪
| 维度 | QA 做了 | 应该做 |
|------|---------|--------|
| 硬件多样性 | CI runner（96 DPI） | 高 DPI、多显示器、不同缩放比 |
| 真实操作系统 | CI 干净环境 | 用户真实桌面（有其他软件、中文路径） |
| 使用场景 | 单命令测试 | 端到端工作流 |
| 输出正确性 | "有输出" = pass | "输出内容对不对" |

## 📋 必须补充的测试维度

### 1. DPI / 高分屏（最大盲区）
- **为什么重要**：2024 年 Steam 硬件调查，超过 35% 用户用高 DPI 屏幕
- **必测场景**：
  - 100% / 125% / 150% / 175% / 200% / 250% 缩放
  - `naturo list screens` 输出是否是 **物理分辨率**
  - `naturo capture live` 截图是否是 **物理分辨率**
  - `naturo see` 的坐标是否是 **物理坐标**
  - `naturo click` 的坐标是否准确（DPI 转换对不对）
  - 多显示器 + 不同 DPI 缩放

### 2. 用户首次安装体验
- `pip install naturo` 然后立刻跑 `naturo --help`
- 随便 copy README 里的第一个例子执行，能不能跑通？
- 报错信息是否友好，还是丢一堆 traceback？

### 3. 真实桌面环境
- 有中文窗口标题
- 有多个显示器
- 有全屏应用
- 有 UAC 弹窗
- 有高 DPI 混合缩放（主屏 150%，副屏 100%）

### 4. 端到端工作流
不是测单个命令，而是测完整场景：
```
场景 1: 截屏并分析
naturo capture live → naturo see → 找到某个元素 → naturo click

场景 2: 监控窗口变化
naturo list windows → naturo wait --title "xxx" → naturo capture window

场景 3: 应用管理
naturo app launch notepad → naturo app find notepad → naturo app close notepad
```

### 5. 输出内容验证（不只看格式）
- `list screens` 的分辨率是否和 Windows 设置一致？
- `capture` 的图片尺寸是否和显示器物理分辨率一致？
- `see` 返回的元素坐标点击后是否真的点到了？
- 不要只检查 JSON 格式合法，要检查 **数值是否正确**

## 🧠 测试思维升级

### 从"功能测试"到"产品质量"
- 功能测试问的是："这个 API 按文档工作吗？"
- 产品质量问的是："用户拿到这个工具，能不能完成他想做的事？"

### 从"命令行参数"到"物理世界"
naturo 不是纯软件工具，它操作的是 **物理屏幕上的像素**。测试必须考虑：
- 硬件差异（分辨率、DPI、显卡驱动）
- 系统配置（缩放、多显示器、主题）
- 进程权限（管理员、UAC、DPI awareness）

### 从"单命令"到"组合使用"
用户不会只跑一个命令。他们会：
1. `see` 看元素 → 2. 拿到坐标/ID → 3. `click` 那个元素
如果 `see` 返回的坐标因为 DPI 问题是错的，`click` 就会点偏。**单独测都 pass，组合起来全挂。**

## ⚡ 立即行动

1. **获取真实 DPI 信息**：SSH 到 Lead 执行 `naturo list screens`，对比实际 Windows 设置
2. **截图尺寸验证**：`naturo capture live`，检查输出图片的像素尺寸 vs 显示器物理分辨率

4. 把以上发现写入 bugs.md

## 🚨 反人性设计 = Bug

**如果用户的第一反应用法不工作，那就是 bug，不管代码逻辑上多"正确"。**

典型案例：
- `see` 输出 `e16`，用户自然会 `click --id e16`，结果报错 → **这是 P0 bug**
- `click --id` 需要 `"Button:最小化"` 格式，但没有任何文档/帮助告诉用户 → **这是 P0 bug**
- 命令帮助说 `--id TEXT`，用户传了 see 给的 ID 却不认 → **这是 P0 bug**

**QA 铁律**：
1. 每个命令先"装傻"试一遍——不看源码，只看 `--help`，按直觉用
2. 如果直觉用法报错 → 报 bug，标 P0
3. 如果需要看源码才能搞清楚正确用法 → 报 bug，至少 P1
4. **反人性的输入要求 = 设计缺陷，和崩溃一样严重**
