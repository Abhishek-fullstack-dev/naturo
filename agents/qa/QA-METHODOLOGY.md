# QA 方法论 — 最佳实践合集

> 这不是"闹呢一条条教你的规则"，而是业界顶级 QA 方法论的精华，适配到 naturo（桌面自动化 CLI 工具）场景。一次到位，不再一个个补。

---

## 一、核心原则

### 1. 最小惊讶原则（Principle of Least Astonishment）
> 软件的行为应该符合用户合理预期。如果用户惊讶了，那就是 bug。

- 用户看到 `e16`，自然会 `click --id e16` → 必须能工作
- `--help` 说接受 `TEXT`，用户传任何文本都不该崩溃
- 命令名叫 `capture`，出来的图应该是屏幕真实样子，不是缩小版
- **检验标准**：任何功能，找一个没看过源码的人试，他的第一反应用法必须成功

### 2. 用户不看文档原则
> 80% 的用户不会读文档。他们会看 `--help`、猜、然后试。

- `--help` 里的例子必须能直接复制运行
- 参数名必须自解释（`--id` 就应该接 see 给的 ID）
- 报错信息必须告诉用户怎么改，不是丢一个 traceback

### 3. 物理世界正确性原则
> naturo 操作的是物理屏幕上的像素，不是抽象 API。测试必须验证物理世界的正确性。

- 分辨率/DPI/坐标 → 和 Windows 系统设置对比，数值必须一致
- 截图 → 和实际屏幕对比，内容和尺寸必须一致
- 点击坐标 → 必须真的点到了目标元素

### 4. 组合正确性原则
> 单个命令 pass ≠ 产品可用。用户使用的是命令组合。

- `see` → `click` → `capture` 是最基础的组合流
- 前一个命令的输出必须能直接作为后一个命令的输入
- 数据在命令间传递时不能丢失精度（坐标、ID、路径）

### 5. 环境多样性原则
> 你的测试环境只代表一种用户。至少覆盖 3 种典型环境。

## 二、测试层次模型

### L0 — 冒烟测试（每次 CI 必跑）
- 安装成功
- `--help` 正常
- 每个命令基本功能 pass
- 退出码正确
- JSON 输出合法

### L1 — 功能测试（每轮 QA 必做）
- 每个参数 × 正常值/边界值/非法值
- 参数组合（互斥、依赖、冲突）
- 错误信息清晰且有指导性
- 文档和行为一致

### L2 — 用户体验测试（每 3 轮做一次）
- **装傻测试**：不看源码，只看 --help，按直觉操作
- **首次体验**：`pip install naturo` → 跑 README 第一个例子
- **命令串联**：see → click → capture → diff 完整流程
- **报错体验**：故意传错参数，报错信息能不能让你改对？

### L3 — 真实环境测试（每 3 轮做一次）
- SSH 到 Lead 真机执行
- 验证物理值（分辨率、DPI、坐标映射）
- 中文窗口标题、中文路径
- 多窗口/多显示器场景
- 高 DPI + 缩放（100%/125%/150%/200%）

### L4 — 竞品对标测试（每个 Phase 做一次）
- 同等功能在 Peekaboo/PyAutoGUI/pywinauto 是什么体验？
- 我们的输出格式是否对 AI Agent 友好？
- 安装/上手门槛对比

## 三、探索性测试启发式（基于 James Bach / SFDPOT）

每次探索性测试，从以下 6 个维度思考：

| 维度 | 问什么 | naturo 举例 |
|------|--------|-------------|
| **S**tructure | 内部结构对输出有影响吗？ | DLL 和 Python 版本不一致、DPI awareness 时机 |
| **F**unction | 功能是否完整、正确？ | click --id 不认 see 的 eN |
| **D**ata | 输入/输出数据对吗？ | 分辨率报告 2560 实际 3840 |
| **P**latform | 环境/平台影响？ | Windows 版本、DPI 设置、权限 |
| **O**perations | 实际使用场景？ | 远程 SSH 截屏、批量操作、长时间运行 |
| **T**ime | 时间相关问题？ | 缓存过期、元素消失后再 click |

### Heuristic Test Strategy Model 检查清单

每个功能问自己：
- [ ] **能力**：它做了它该做的事吗？
- [ ] **可靠性**：每次都能做到吗？不同环境下呢？
- [ ] **易用性**：用户能不看文档就会用吗？
- [ ] **性能**：够快吗？资源消耗合理吗？
- [ ] **可安装性**：pip install 一步搞定？依赖清楚？
- [ ] **兼容性**：Win10/Win11？不同 Python 版本？
- [ ] **可维护性**：报错信息有助于排查问题吗？

## 四、Bug 严重度标准

### P0 — 阻断（必须当天修）
- 核心功能不工作（截图、点击、see）
- 用户第一直觉用法失败
- 输出数据错误（分辨率/坐标/DPI 不对）
- 崩溃 / 未捕获异常

### P1 — 严重（本轮修）
- 非核心功能不工作
- 报错信息无用（traceback / 看不懂）
- 文档和行为不一致
- --help 例子跑不通

### P2 — 一般（下轮修）
- 边界值处理不当
- 输出格式不一致
- 性能问题

### P3 — 改进（排期修）
- UX 可优化但不影响功能
- 代码风格 / 命名建议

## 五、反模式（QA 绝对不能犯的错）

### ❌ 只在 CI 环境测
CI runner 是 96 DPI、干净系统、英文。用户不是。

### ❌ 只测"能不能跑"不测"对不对"
`list screens` 有输出 ≠ 输出正确。必须和物理世界对比。

### ❌ 只测单命令不测组合
`see` pass + `click` pass ≠ `see → click` 能串起来。

### ❌ 看了源码才知道怎么用
如果你需要看源码才能正确使用某个功能 → 那个功能的 UX 有 bug。

### ❌ 发现问题后只报功能 bug，不追问设计缺陷
"click --id 不认 e16" 不只是一个 bug，是 `see` 和 `click` 的设计没打通。要追到根因。

### ❌ 只关注新功能，忽略回归
改了 DPI 逻辑 → 所有涉及坐标的命令都要重测。

## 六、QA 每轮自检清单

每轮结束前回答：

1. 本轮有做 L2（用户体验）测试吗？
2. 最近 3 轮有做 L3（真机）测试吗？
3. 有没有"装傻"试过每个新功能？
4. 发现的 bug 有没有追到设计层面？
5. 测试结果中，有多少是验证"对不对"而不只是"能不能跑"？
6. 如果一个新用户现在 pip install naturo，他会满意吗？

**如果第 6 题答案是否 → 找到原因 → 报 P0。**
